# ADR-004: Activity Registry

## Status

Proposed

## Context

The wtf-engine needs a central registry for managing activity handlers. Activities are the executable units within a workflow DAG. The registry must support:
- Dynamic registration/unregistration of activities
- Versioned activity names
- Timeout and retry semantics for activity execution
- Built-in system activities

## Decision

### Data Structures

```rust
pub struct ActivityHandler {
    name: String,
    version: String,
    handler: Arc<dyn Fn(Vec<u8>) -> BoxFuture<Result<Vec<u8>, ActivityError>> + Send + Sync>,
    description: Option<String>,
}

pub struct ActivityRegistry {
    activities: HashMap<String, ActivityHandler>,
}
```

### Key Design Decisions

1. **Name-version keying**: Activities keyed by `"name:version"` for version isolation
2. **Arc-wrapped handlers**: Enable sharing across async tasks
3. **BoxFuture return**: Async handler support with error propagation
4. **Exponential backoff retry**: Built-in retry with configurable base delay
5. **Timeout wrapper**: Per-execution timeout enforcement

### Activity Execution Flow

1. Lookup activity handler by name
2. Apply timeout wrapper
3. Execute handler
4. On failure, apply retry with exponential backoff
5. Return result or final error

### Built-in Activities

- `sleep`: Pauses execution for specified milliseconds
- `wait_for_signal`: Blocks until external signal received

## Consequences

- Registry is synchronous; async operations via `execute()` method
- Retry state tracked per-execution, not globally
- Thread-safe handler storage via Arc
